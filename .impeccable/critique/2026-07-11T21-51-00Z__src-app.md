The strongest opportunity is not visual novelty; it is making the interaction model as coherent as the visual system.

## Overall Impression

This is a calm, coherent authenticated shell with a good information architecture. The change improves orientation substantially, especially on mobile. However, the navigation contract is only partially implemented: the most important exits are protected while equivalent exits are not, and Practice currently hydrates with a runtime error. The single biggest opportunity is to make every route exit obey one navigation/dirty-state policy.

## What's Working

- The shared header and breadcrumb establish a stable location model without adding a heavy navigation layer.
- The visual language is disciplined: one primary action, neutral surfaces, readable type, touch-sized controls, and no detector-detected design slop.
- Form copy and structure are clear, with explicit labels, inline card validation, Cancel actions, and useful Save-and-add-another behavior.

## Priority Issues

### [P1] Dirty-form protection is incomplete across equivalent exits

**Why it matters:** `Breadcrumb` renders raw `next/link` elements at `src/components/app/breadcrumb.tsx:65`, while Cancel and the header use `GuardedLink`. A user can edit a deck/card, click the breadcrumb parent, and leave without confirmation. Browser back and mobile back gestures are also not intercepted: `use-unsaved-changes-warning.ts` registers only `beforeunload` at lines 24–25, which does not cover App Router history navigation.

**Fix:** Make breadcrumb ancestor links use the same guarded navigation primitive and add a tested App Router/popstate interception strategy. Keep `beforeunload` for hard exits, but do not treat it as the soft-navigation guard.

**Suggested command:** `$impeccable harden`

### [P1] Practice has a hydration mismatch on first render

**Why it matters:** `StudySession` calls `Math.random()` during render through `shuffle()` at `src/components/study/study-session.tsx:39`, so server and client produce different card orders. The browser showed a Next.js issue badge and `Hydration failed because the server rendered text didn't match the client` on `/study?mode=practice`.

**Fix:** Make the initial order deterministic across server and client, or defer client-only shuffling until after hydration with a stable loading/transition state. Add a browser or component regression test that asserts no hydration error for Practice.

**Suggested command:** `$impeccable harden`

### [P1] `router.back()` has no safe fallback for direct-entry edit pages

**Why it matters:** `DeckForm` and `FlashcardForm` call `router.back()` at `src/components/decks/deck-form.tsx:41` and `src/components/cards/flashcard-form.tsx:99`. This works when the user arrived from the deck, but a refreshed/bookmarked/new-tab edit URL can send the user to an unrelated previous page or outside the app after saving. The implementation is history-relative, not parent-relative.

**Fix:** Preserve the back-button optimization for known in-app edit flows, but provide an explicit parent fallback for direct entry and verify the destination belongs to the app before leaving. Add tests for a fresh edit URL with no parent history.

**Suggested command:** `$impeccable harden`

### [P2] Client-side save refactor weakens local failure recovery

**Why it matters:** Edit forms now invoke Server Actions inside `startTransition`/a form action and immediately call `router.back()` on resolution. There is no local error state or retry path if the action fails after the request starts; the user can receive only a global error boundary or a console-level failure. Pending labels communicate activity but not failure or success.

**Fix:** Use an action-state/error boundary pattern that preserves field values, explains the failure inline, and keeps the user on the form. Show a concise success confirmation before or during the return transition if the navigation is not immediate.

**Suggested command:** `$impeccable harden`

### [P2] Desktop composition is too narrow for the authenticated shell

**Why it matters:** `AppScreen` fixes every authenticated view to `max-w-md` at `src/components/app-screen.tsx:26`. At a desktop viewport the content occupies a small centered column with a large amount of unused space, while Deck detail still has a long card list. The mobile-first constraint is good; the desktop layout does not adapt structurally.

**Fix:** Keep the compact reading width for forms and Study, but allow list/detail screens to widen at a breakpoint (for example, a modest `max-w-2xl`) and let the page-specific surface choose its density.

**Suggested command:** `$impeccable adapt`

## Cognitive Load

**1 checklist failure — low load overall.** The shell reduces memory load with visible breadcrumbs and progressive disclosure for deck/card actions. The main remaining failure is inconsistent navigation behavior: users must remember that Cancel and the logo warn, while breadcrumbs and browser back may not. The Deck detail page also combines study actions, add-card, deck actions, and a long card list, but the primary Study action remains clear.

## Emotional Journey

- **Start:** Home is calm and immediately tells the user what to study next.
- **Orientation:** Breadcrumbs reduce uncertainty and make the current deck/card context legible.
- **High-stakes moment:** Editing feels safe when using Cancel or the logo, but unsafe when using the breadcrumb or mobile back gesture because the warning is inconsistent.
- **Completion:** Returning to the parent after save is efficient, but the abrupt return without a saved confirmation weakens reassurance.
- **End:** Study completion has a clear “Back to [deck]” action, but Practice first needs the hydration mismatch removed to preserve trust.

## Persona Red Flags

### Sam — Accessibility-dependent user

- The soft-navigation dirty guard is not reliable for keyboard/browser-history paths; Sam can lose edits without a prompt.
- Practice exposes a hydration error badge in development, indicating an unstable render path.
- The visible three-dot action menus are accessible by name, but their available actions remain hidden until activation; the menu must retain clear focus and state announcements.

### Casey — Distracted mobile user

- The mobile breadcrumb provides a useful parent path, but the same back gesture can bypass unsaved-change protection.
- There is no draft persistence if Casey leaves the app or is interrupted while editing.
- Touch target sizing is generally strong, but the safety model depends too much on knowing which links are guarded.

### Riley — Deliberate stress tester

- A bookmarked or freshly opened edit URL can make `router.back()` return to an unrelated history entry after save.
- Practice produces a hydration mismatch because its initial random order differs between server and client.
- The refactored client submission path needs explicit retry behavior for server failures instead of relying on a global error surface.

## Minor Observations

- `breadcrumb-context.ts` comments say Home is implicit and prepended, but callers pass Home explicitly; the documentation is misleading.
- `Breadcrumb` falls back to `href="#"` for a non-current item without an `href`; the type and renderer should make this impossible or render non-clickable text.
- The desktop visual hierarchy is otherwise strong, but the `Deck` eyebrow and all-caps Study labels are slightly more editorial than the otherwise quiet product vocabulary.
- The browser detector's font finding should be ignored for this project; a single system sans is intentional.

## Questions to Consider

- Should every internal route exit—including breadcrumbs and browser/mobile back—share one unsaved-change guard, even if that requires a small custom history manager?
- Is preserving `router.back()` worth the direct-entry complexity, or should save always resolve to an explicit parent route for predictability?
- Should the authenticated shell widen on desktop for deck/card lists while keeping forms and Study compact?
